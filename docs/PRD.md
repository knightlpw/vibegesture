# PRD.md — VibeGesture

## 1. 产品概览

### 1.1 产品名称
VibeGesture

### 1.2 产品定位
VibeGesture 是一款轻量级的 macOS 菜单栏工具，面向 vibe coder。它使用 Mac 默认摄像头识别一组固定的右手手势，并将其转换为键盘动作，用于语音输入驱动的编程工作流。

### 1.3 核心价值
VibeGesture 通过手势替代重复性的键盘操作，帮助用户在 vibe coding 过程中减少摩擦，保持更顺畅的流程。

典型动作包括：
- 开始语音输入
- 停止语音输入
- 提交文本
- 取消当前动作

### 1.4 目标用户
经常使用以下工具的用户：
- Codex
- Claude Code
- Cursor

这些用户会在说话、等待转写、提交提示词和取消输入之间频繁切换。

---

## 2. 问题陈述

在 vibe coding 工作流中，用户通常会依赖 Typeless 等语音输入工具来口述提示词或与代码相关的指令。

即使已经使用语音输入，用户仍然需要手动按一些键盘快捷键，例如：
- 单击某个键开始录音
- 再单击同一个键停止录音
- 按 Enter 提交
- 按 Esc 取消

这些重复性的键盘交互会引入摩擦，并打断免手操作或低摩擦编码的节奏。

市面上也有一些基于硬件的方案，例如用于语音输入、提交和取消动作的迷你键盘，但硬件会带来成本、便携性问题和额外的设置复杂度。

VibeGesture 的目标是用摄像头手势识别提供一个更轻量的软件方案。

---

## 3. 产品目标

### 3.1 主要目标
让用户仅通过手势完成一个基础的语音输入编码闭环：

1. 打开手势识别
2. 用 record 手势一次开始语音输入
3. 再用 record 手势一次停止语音输入
4. 用 submit 手势提交文本
5. 用 cancel 手势中止当前动作

### 3.2 成功标准
MVP 被认为成功的条件是：
- 端到端手势流程在支持的应用中可用
- 误触发率足够低，适合短时间实际使用
- 感知延迟足够低，不会打断输入和说话节奏
- 权限引导和手势开关足够易懂

### 3.3 非目标
本产品在 V1 中**不**追求：
- 成为通用手势自动化平台
- 支持任意应用
- 支持自定义手势训练
- 支持左手识别
- 支持双手同时识别
- 直接集成 Typeless 内部机制
- 验证转写文本是否已经进入输入框
- 上架 Mac App Store
- 永久在线后台监控
- 在 V1 中支持用户可编辑的应用白名单

## 4. 产品范围

### 4.1 平台范围
- 仅支持 macOS
- 仅通过网站下载安装包分发
- 仅做菜单栏工具

### 4.2 用户范围
- 明确面向 vibe coder
- 主要服务于 Codex、Claude Code 和 Cursor 用户

### 4.3 摄像头范围
- 仅使用默认摄像头
- 仅支持右手
- 仅支持单手识别

### 4.4 手势范围
V1 仅支持以下固定、由用户触发的手势：
- `record`
- `submit`
- `cancel`

Record exit / re-arm 是内部检测条件，不是面向用户的手势，也不是键盘事件。

不支持自定义手势。  
不支持手势训练。  
不支持用户自定义手势集合。

### 4.5 手势与系统操作映射
| 手势 | 面向摄像头的手部动作 | 系统操作 | 说明 |
| --- | --- | --- | --- |
| Record | 右手拇指和食指捏合，其他手指握拳 | `Fn` 单击 | 录音开 / 关切换 |
| Submit | 右手呈 `OK` 状，拇指和食指捏合，其他三指张开 | `Enter` 单击 | 录音开启时先停录，再发送 Enter |
| Cancel | 右手手掌张开，掌心面对摄像头 | `Esc` 单击 | 直接发送 Esc |

以上三种手势是 V1 唯一需要明确识别并映射到系统操作的动作。
Record exit / re-arm 仍然只是内部重新武装条件，不是用户可见动作。

## 5. 核心使用场景

### 5.1 主要流程
1. 用户打开 Codex、Claude Code 或 Cursor
2. 用户通过菜单栏开关或全局快捷键启用 VibeGesture
3. VibeGesture 开始基于摄像头的手势识别
4. 用户执行 record 手势
5. VibeGesture 单击一次配置好的录音键，开始录音
6. 用户说话
7. 用户再次执行 record 手势
8. VibeGesture 再单击一次配置好的录音键，停止录音
9. 用户执行 submit 手势
10. VibeGesture 发送 Enter
11. 用户继续工作

### 5.2 取消流程
在识别开启期间，用户可以随时执行 cancel 手势：
- 如果录音处于开启状态，VibeGesture 会先单击录音键一次，停止录音
- VibeGesture 再发送 Esc
- 当前手势动作会被中断
- 系统返回 idle 识别状态

## 6. 支持的应用

### 6.1 前台应用限制
只有当前前台应用属于支持列表时，手势动作才会生效。  
如果前台应用变成不支持的应用，而此时录音仍在进行中，VibeGesture 会先单击录音键一次停止录音，然后再抑制后续手势动作。

### 6.2 V1 默认支持的应用
- Codex
- Claude Code
- Cursor

### 6.3 白名单行为
- 白名单在 V1 中是固定的
- 用户在 V1 中不能编辑白名单
- 如果前台应用不受支持，则忽略手势
如果前台应用变为不受支持，而录音仍然开启，VibeGesture 会先停止录音，然后在支持的应用重新出现之前忽略手势

### 6.4 应用检测
前台应用匹配应基于 bundle identifier。

## 7. 功能需求

## 7.1 识别开关

### 需求
- 用户可以通过菜单栏启用或禁用手势识别
- 用户可以通过全局快捷键启用或禁用手势识别
- 手势识别在 1 小时的 active recognition time 之后自动关闭
- 手动关闭识别会重置下一次激活的 1 小时计时
- App 默认不在登录时自动启动
- App 不会在跨启动之间记住上一次的识别开关状态

### 预期行为
- 开启时，菜单栏图标变色
- 关闭时，图标恢复为非激活外观
- 切换开关时会出现一个 toast / floating overlay
- 如果识别被关闭、超时关闭，或者在录音期间丢失了必要权限，VibeGesture 会先单击录音键一次停止录音，再完全禁用识别

---

## 7.2 手势：Record

### 定义
Record 是一个一次性 toggle 手势，用于开始或停止语音录音。
Record 的面向摄像头动作是：右手拇指和食指捏合，其他三指握拳。

### 触发行为
当 `record` 进入 active 状态且 cooldown 不处于激活状态时：
- 单击一次配置好的录音键

### 默认映射
- 默认录音键：`Fn`

### 可配置性
- 用户可以在设置中把录音键改成另一个单键快捷键

### 重复行为
- 持续保持 record 不会重复触发录音键
- 若要再次触发，用户必须离开 record，满足 record re-arm 条件，再重新进入 record

### 录音语义
- 如果当前录音是关闭的，record 会开始录音
- 如果当前录音是开启的，record 会停止录音

---

## 7.3 手势：Record Exit / Re-arm

### 定义
Record exit 不是用户可见的手势。  
它只是内部条件，用来让系统为下一次 record 触发重新武装。

### 触发行为
当 record 连续达到 record re-arm 阈值时：
- 不发送任何键盘事件
- 系统变为可以接受下一次 record 触发的状态

### 说明
Record exit 应被视为内部检测转移，而不是产品动作。

---

## 7.4 手势：Submit

### 定义
Submit 是一个一次性手势，用于发送 Enter。
Submit 的面向摄像头动作是：右手呈 `OK` 状，拇指和食指捏合，其他三指张开。

### 触发行为
当 `submit` 进入 active 状态时：
- 如果当前录音关闭：
  - 发送 `Enter`
- 如果当前录音开启：
  1. 单击一次录音键，停止录音
  2. 等待 300 ms
  3. 发送 `Enter`

Submit 的行为由采样到 submit 手势时的录音状态决定。

### 重复行为
- Submit 只会在一次激活中触发一次
- 持续保持 submit 不会重复发送 Enter
- 用户必须离开 submit 手势再重新进入，才会再次触发

### 默认映射
- `Enter`

### 可配置性
- V1 中 Submit 键可以在设置中配置

---

## 7.5 手势：Cancel

### 定义
Cancel 是一个一次性手势，用于中止当前动作。
Cancel 的面向摄像头动作是：右手手掌张开，掌心面对摄像头。

### 触发行为
当 `cancel` 进入 active 状态时：
- 发送 `Esc`
- 中断当前手势流程
- 返回 idle 识别状态

### 优先级
- Cancel 在所有手势中优先级最高
- 如果 confidently 检测到 cancel，它应该覆盖其他状态
- 如果在 submit 的延迟窗口内检测到 cancel，则 cancel 会抑制后续 Enter

### 默认映射
- `Esc`

### 可配置性
- V1 中 Cancel 键可以在设置中配置

---

## 7.6 权限

### 必需权限
- Camera 权限
- Accessibility 权限

### 首次启动行为
- 展示权限引导流程
- 解释为什么需要这些权限
- 如果缺少权限，优先引导用户打开缺失的那个权限
- 如果两个权限都缺失，则一次只引导一个权限

### 缺失权限处理
- 必需权限缺失时，识别不能启动
- 当权限缺失时，要展示清晰的 UI 提示
- 如果权限在录音期间被撤销，VibeGesture 会先单击录音键一次停止录音，然后进入权限错误状态

---

## 7.7 反馈与 UI

### 菜单栏反馈
- 非激活状态：中性图标
- 激活识别状态：变色图标

### 浮层反馈
当动作被触发时，在屏幕上方偏中间位置显示一个小型 floating overlay。

### 默认浮层文案
- `Gesture On`
- `Gesture Off`
- `Recording`
- `Recording Stopped`
- `Submitted`
- `Cancelled`
- `Permission Required`

### UI 原则
- 轻量
- 不打扰
- 快速
- 足够清晰，减少不确定性

## 8. 状态机设计

## 8.1 内部状态
V1 建议的内部状态：
- `disabled`
- `idle`
- `recording_active`
- `cooldown`
- `error_permission_missing`

### 状态说明
- `disabled`：手势识别关闭
- `idle`：识别开启，录音关闭，系统正在等待有效手势
- `recording_active`：目标应用中的录音当前处于开启状态
- `cooldown`：一次性动作后的短暂临时状态，用于防止重复触发；cooldown 期间不处理任何手势，结束后系统会回到 `idle` 或 `recording_active`
- `error_permission_missing`：所需权限未授予

---

## 8.2 状态转移

### disabled -> idle
条件：
- 用户开启识别
- 必需权限存在
- 前台应用受支持

动作：
- 开始手势识别
- 确保录音未处于开启状态

### idle -> cooldown
条件：
- Record 被稳定识别

动作：
- 单击一次录音键，开始录音
- 显示 `Recording`

### cooldown -> recording_active
条件：
- Record-start cooldown 结束

动作：
- 进入 recording-active 模式

### recording_active -> cooldown
条件：
- Record 被稳定识别

动作：
- 单击一次录音键，停止录音
- 显示 `Recording Stopped`

### idle -> cooldown
条件：
- submit 或 cancel 触发

动作：
- 发送相应键盘事件
- 显示对应浮层

### recording_active -> cooldown
条件：
- 在录音开启时触发 submit

动作：
1. 单击一次录音键，停止录音
2. 等待 300 ms
3. 发送 Enter
4. 显示 `Submitted`

### recording_active -> cooldown
条件：
- 在录音开启时触发 cancel

动作：
1. 单击一次录音键，停止录音
2. 发送 `Esc`
3. 显示 `Cancelled`

### cooldown -> idle
条件：
- 停录、submit 或 cancel 的 cooldown 结束

动作：
- 进入 idle 模式

### any active state -> disabled
条件：
- 用户手动关闭识别
- 或识别在 1 小时后自动超时关闭

动作：
- 如果录音处于开启状态，先单击录音键一次停止录音
- 显示 `Gesture Off`

### any state -> error_permission_missing
条件：
- 必需权限缺失或被撤销

动作：
- 如果录音处于开启状态，先单击录音键一次停止录音
- 显示 `Permission Required`

---

## 8.3 时间默认值
以下数值是 MVP 调优的默认值，后续实现时可以微调：

- 识别目标帧率：`10–15 FPS`
- record 激活阈值：`6 consecutive frames`
- record re-arm 阈值：`4 consecutive frames without record`
- submit 激活阈值：`4 consecutive frames`
- cancel 激活阈值：`3 consecutive frames`
- 一次性 cooldown 持续时间：`700 ms`
- submit-while-recording-stop 延迟：`300 ms`
- 识别会话超时：`1 hour`

## 9. 技术方案

## 9.1 检测策略
V1 应使用：
- Apple Vision hand pose detection
- 基于规则的手势解释
- 基于 state machine 的事件发射

### 明确的技术约束
V1 **不应**使用：
- 自定义训练好的手势模型
- 用户专属的手势训练流水线
- 复杂的机器学习基础设施

### 设计理由
MVP 只需要一个很小的固定手势集合，因此应该优先考虑：
- 实现简单
- 产品风险低
- 行为可预测
- 迭代速度快

---

## 9.2 事件发射
产品不直接集成 Typeless 的内部机制。  
它只向系统发送键盘事件。

### 原则
VibeGesture 本质上是一个：
**gesture-to-key-event mapper**

这让应用更通用，同时仍然能够支持目标 Typeless 工作流。

V1 发射的是单键 tap，而不是按住/松开式的 press and release 序列。

## 10. 设置

## 10.1 V1 设置范围
V1 的最小设置项：
- 启用 / 禁用手势识别
- 配置全局 toggle 快捷键
- 配置 record toggle 快捷键
- 配置 submit 快捷键
- 配置 cancel 快捷键

V1 中 record toggle 快捷键仅允许单键。  
设置入口应是一个轻量级的菜单项 popover；如果实现需要，也可以接受一个小型系统设置窗口。

### V1 不包含
- 应用白名单编辑
- 手势自定义
- 左右手切换
- 识别灵敏度调节 UI
- 高级自动化逻辑

## 11. UX 要求

### 11.1 轻量体验
应用必须感觉轻量且即时：
- 启动快
- UI 装饰少
- 打断少
- 状态清晰

### 11.2 可预测性
用户应该始终知道：
- 识别是否开启
- 录音是否处于开启状态
- 某个动作是否已经触发
- 是否是权限阻塞了系统

### 11.3 安全性
系统应优先忽略不明确的手势，而不是做出意外动作。  
当录音处于开启状态时，在终端类动作前先停止录音，比让目标应用继续保持录音状态更安全。

## 13. 风险

### 13.1 误触发风险
手势混淆或识别不稳定可能导致意外动作。

缓解措施：
- 保守阈值
- cooldown 窗口
- 固定手势集合
- 前台应用限制
- 清晰的视觉反馈
- record re-arm gating，避免同一次 record 被触发两次

### 13.2 延迟风险
识别延迟过高会破坏用户流程。

缓解措施：
- 足够低但合理的帧率
- 简单的规则分类
- 尽量短的内部流水线

### 13.3 权限摩擦
Camera 和 Accessibility 权限对用户来说可能感觉比较重。

缓解措施：
- 清晰的 onboarding
- 简洁的权限说明
- 一步一步引导用户

### 13.4 Typeless 时序不确定性
应用无法可靠知道转写是否已经进入输入框。

缓解措施：
- 保持逻辑简单
- 只发送键盘事件
- 在 submit-while-recording-stop 场景使用安全延迟
- 不假装去读取外部应用内部状态

## 17. 最终 MVP 定义

VibeGesture 的 V1 在满足以下条件时就算完成：

1. 它可以作为 macOS 菜单栏应用运行
2. 它可以通过菜单栏和全局快捷键切换开启 / 关闭
3. 它会请求并处理 Camera 和 Accessibility 权限
4. 它只在 Codex、Claude Code 或 Cursor 位于前台时工作
5. 它会使用默认摄像头识别一组固定的右手手势
6. 它的映射如下：
   - record -> 通过录音键单击切换录音
   - submit -> Enter
   - cancel -> Esc
7. 它会显示菜单栏和 floating overlay 反馈
8. 它会在 1 小时 active recognition time 后自动关闭，并在需要时先停止录音
9. 整个工作流在实践中可用，延迟可接受，误触发率足够低
