本轮只完成：搭建手势解释与显式 state machine 骨架。

---

## 本轮背景

- TASK001 已完成并验证通过
- TASK002 已完成并验证通过
- TASK003 已完成并验证通过
- 当前仓库已经有可运行的 macOS 菜单栏 shell、配置骨架、权限状态、默认摄像头采集、Vision hand pose 观测骨架与摄像头 / pipeline 状态展示
- 当前实现仍然没有 gesture interpreter、recognition state machine、keyboard dispatcher 或前台应用 gating
- roadmap 的第 4 阶段是手势解释与 state machine
- 下一步应先把原始观测变成可测试的 gesture candidate 和显式状态机，而不要提前接键盘发射

---

## 本轮要做

- 新增 gesture interpreter，从 `CameraFrameObservation` 中提取 pinch / submit / cancel / re-arm 候选信号
- 新增 recognition coordinator / state machine，集中管理 `disabled` / `idle` / `recording_active` / `cooldown` / `error_permission_missing`
- 把 pinch 的 re-arm / cooldown 规则做成明确的内部状态转换
- 让 submit / cancel 变成一次性候选事件，保留后续第 5 阶段接键盘发射的接口
- 在 AppState / 设置窗口里展示当前 gesture / recognition 状态，便于调试
- 先补测试，保证阈值和状态迁移可被稳定验证

---

## 本轮不要做

- 不要发 keyboard event
- 不要做 frontmost app gating
- 不要做 overlay / floating feedback
- 不要把设置页面扩成完整偏好设置
- 不要改摄像头或 Vision pipeline 本身
- 不要把 pinch 重新做回按住 / 松开语义
- 不要把 gesture interpreter 分散到多个 view model

---

## 本轮主流程
1. 读取 camera / Vision 观测
2. 归一化 pinch / submit / cancel / re-arm 候选
3. 由单一 state machine 决定当前识别态与下一步动作意图
4. 将状态与最近一次候选结果暴露给 AppState / UI
5. 用测试覆盖关键转移与 cooldown 行为

---

## 建议实现内容

以下不是唯一实现方式，但请保持目标一致：

- Swift + 现有 AppCoordinator / AppState 结构继续扩展
- 解释器与 state machine 保持在单一串行协调器 / 队列上运行
- 阈值沿用 PRD / TECH 文档：pinch 6 帧、re-arm 4 帧非 pinch、submit 4 帧、cancel 3 帧、cooldown 700ms、submit-stop 300ms
- 解释器只产出内部 intent，不直接触发键盘行为
- 后续 keyboard dispatcher 直接消费这些 intent，不再重新解释原始帧流

---

## 验收标准

- [ ] App 能从摄像头 / Vision 观测中产出 pinch / submit / cancel 候选
- [ ] state machine 的核心状态是显式的，不依赖散落在 view 里的条件判断
- [ ] pinch 的 cooldown / re-arm 行为可重复验证
- [ ] submit / cancel 的候选触发是一次性的，不会持续重复触发
- [ ] 当前识别态可以在 UI 中被看见，便于调试
- [ ] `swift build` 和 `swift test` 仍然通过

- 优先保守实现，不自行扩展需求

- 若遇到未定义需求，默认不实现
