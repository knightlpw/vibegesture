## 本轮目标
本轮只完成：搭建键盘事件发射与安全时序骨架。

---

## 本轮背景

- TASK001 已完成并验证通过
- TASK002 已完成并验证通过
- TASK003 已完成并验证通过
- TASK004 已完成并验证通过
- 当前仓库已经有可运行的 macOS 菜单栏 shell、配置骨架、权限状态、默认摄像头采集、Vision hand pose 观测骨架、gesture interpreter、显式 recognition state machine 与 recognition 状态展示
- 当前实现仍然没有真正的键盘事件发射、前台应用 gating 或 overlay
- roadmap 的第 5 阶段是键盘事件发射与安全时序
- 下一步应把 recognition action intent 稳定转成键盘动作，并保证 submit / cancel 的时序安全

---

## 本轮要做

- 新增集中式 keyboard dispatcher
- 把 `RecognitionActionIntent` 转成真实键盘事件
- 支持 record toggle 的单键 tap，且不对 `Fn` 做特殊语义分支
- 支持 submit 的安全时序：录音关闭时直接发 submit；录音开启时先停录，再等待 300 ms，再发 submit
- 支持 cancel 的安全时序：录音关闭时直接发 cancel；录音开启时先停录，再发 cancel
- 让 cancel 可以抢占并清除待发送的 submit 动作
- 保留后续 app gating 所需的接口，但不在本轮实现前台应用白名单判断

---

## 本轮不要做

- 不要做前台应用 gating
- 不要做 overlay / floating feedback
- 不要把设置页面扩成完整偏好设置
- 不要改摄像头或 Vision pipeline
- 不要改 gesture interpreter 或 state machine 的规则
- 不要绕过 dispatcher 直接发键盘事件

---

## 本轮主流程
1. 由 recognition coordinator / AppCoordinator 接收 action intent
2. 交给单一 keyboard dispatcher 负责实际键盘事件
3. submit-stop 的 300 ms 延迟由 dispatcher 内部的安全时序管理
4. cancel 到达时可以清除待发送的 submit
5. 在 UI / 状态展示中保留最后一次动作结果，便于验证

---

## 建议实现内容

以下不是唯一实现方式，但请保持目标一致：

- Swift + 现有 AppCoordinator / RecognitionCoordinator 结构继续扩展
- keyboard dispatcher 用独立对象封装，避免 UI 层和识别层直接发键盘
- dispatcher 保持串行语义，确保 submit 延迟、cancel 抢占和 stop-first 顺序可预测
- record toggle 只允许单键配置，按普通快捷键事件处理
- submit / cancel 可直接消费现有配置中的快捷键定义
